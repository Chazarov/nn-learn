## Чего нет или что «сырое» (главные пробелы)

1. **Нет включения Кохонена в приложение**: в `container.py` и `ports/api/routes.py` нет сервисов/роутов для сети или карты Кохонена — пользователь **не может** через API обучить сеть или получить карту.
2. **Таблица БД для Кохонена, скорее всего, не создаётся**: в `main.py` импортируется только `models.db_models`; `KohonenProjectDB` туда не подключён, значит `create_all` **не зарегистрирует** `kohonen_projects` (если нет другого импорта).
3. **Репозитории Кохонена сейчас неконсистентны с остальным кодом**: импорт `from models.project import ...` — файла `models/project.py` в дереве **нет**; у `KohonenDiskRepo` поле `clasters`, тогда как в `progect_nn.NNData` используется `classes` — такой слой сохранения **не согласован** с текущими Pydantic‑моделями.
4. **Нет цельного цикла обучения**: нет эпох, нет явного **затухания** \(\alpha(t)\), \(\sigma(t)\) (в `WeightApdator` `sigma` захардкожен как `1.0`), нет критерия останова по ошибке квантования, нет **мёртвых нейронов** / потенциалов.
5. **Топология 2D карты m×n**: калькуляторы топологии сейчас **1D** (индекс нейрона по строке); для произвольной сетки \(m×n\) и корректного соседства на решётке это ещё не доведено.
6. **Победитель в `WeightApdator`**: используется `np.argmax(output_vector)`. Если в `output_vector` лежат **расстояния** (как в `EuclideanVectorDistanceCalculator` — сумма квадратов), победитель должен быть **`argmin`**, иначе логика обратная. Это нужно поправить при сборке полного конвейера.
7. **Прикладная часть задания** (п. 2–4, 6–7, 9–10): выбор предметной области, кодирование номинальных/порядковых признаков, отчёт и эксперименты — в репозитории **не отражены** как готовый результат, только общие CSV и текст методички.
